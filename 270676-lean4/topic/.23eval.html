Henrik Böving (Jul 16 2022 at 16:17):

You're confusing the meta level with the actual language level here. You have made a thing of type Expr which (in this case) contains the representation of a meta variable the Lean compiler uses to operate on them internally. It is not a meta variable that you can actually play with from the Lean user code right now (thought the MetaM and Tactic chapter will demonstrate how this is possible)

So what #eval is doing here is simply showing you how the meta variable representation you've built looks to be precise it shows (in order):

  1. The Lean internal name
  2. The meta data consisting of:
    1. its hash (this is computer science stuff dont worry about it)
    2. a variable which (as its name suggests) says whether the represented expression contains a meta variable (it obviously does, after all it is one)
    3. the binder information which is related to what binder this thing comes from, whether its implicit, instance parameter, or just a default variable, see here for the variants: docs4#Lean.BinderInfo its also not too interesting for you

both the type and the userName you gave to mkFreshExprMVar do not appear here because they are stored in the meta variable context which is part of the state MetaM keeps for you and is thus hidden from you right now

Henrik Böving (Jul 16 2022 at 16:28):

The #eval is also using a VM here, you can either evaluated pure expressions with it: #eval 1 + 1 or certain monadic ones that #eval knows how to run, this is among others IO and MetaM. Once you have evaluated the expression via the VM Lean will use docs4#ToString or docs4#Repr instances in order to pretty print the thing that you have produced (if these aren't available #eval will refuse to run)
